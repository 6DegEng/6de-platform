## Deferred TODOs

1. **Calendar view — `streamlit-calendar` integration.** The current
   Calendar view (`render_calendar_view` in `1_Projects.py:1186-1208`)
   is a clean placeholder: `st.info("Calendar view — coming up next.")`
   plus a metric counting target_end_dates falling in the current
   calendar month. The placeholder renders without crashing and surfaces
   useful information. The full `streamlit-calendar` integration
   (month grid, events colored by status, drill-down to project) is
   estimated at ~30 minutes including a regression test. **Decision:**
   I did NOT ship the upgrade in this session — the goal contract said
   "may ship as preview," and four other views need Gate 4 eyeballing
   before adding a fifth. **File for Session 3a.1 or fold into 3b's
   info-capture work.**

2. **Real cross-column Kanban drag-and-drop.** The scout's recommended
   `streamlit-sortables` library only accepts `list[str]` items, which
   precludes the rich-card layout the prompt requires (bold job number,
   truncated name, client / target-close captions, View button). The
   shipped fallback — a per-card status selectbox — produces the same
   outcome (project moves columns) and routes through `update_project`
   so the activity log captures the move as a `status_changed`
   semantic event. **A custom Streamlit component (React + iframe
   bridge) is the only path to true DnD**, scoped at ~1 day, low
   priority since the selectbox UX is functionally complete.

3. **Plotly bar-click-to-open-project.** Plotly click events under
   Streamlit are unreliable (the `streamlit-plotly-events` shim is
   stale, last commit 2023). The shipped fallback is a below-chart
   selectbox that flips `ui:projects:focus`. Same UX outcome — one
   extra click. Real bar-click event handling needs custom JS bridge
   work; defer until/if `streamlit-plotly-events` revives or Streamlit
   ships native event hooks.

4. **Latent test-fixture bug — env-var-after-import.** The
   `PLATFORM_DB_PATH` env var that
   `tests/test_projects_search.py::app_test` and
   `tests/test_3a_e2e_smoke.py::projects_page` set via
   `monkeypatch.setenv` is read at `config` module import time. By the
   time `AppTest.from_file` triggers a `config` reload inside its
   subprocess, the env var IS in effect — so the tests do hit the
   isolated DB inside the script context. But the surrounding
   pytest-side imports of `config` and `db` may already have resolved
   the production DB path, which is what the 3 rewritten AppTest tests
   in `test_projects_search.py` defensively handle by reading
   `config.DB_PATH` post-hoc. **Not a regression** — same fixture
   pattern from before Session 3a; flagged here so a future cleanup
   session can hoist the config import inside the fixture and
   guarantee the page renders against the seeded DB.

5. **`datetime.utcnow()` deprecation.** `modules/projects/crud.py:21`
   emits 31 DeprecationWarning rows in this session's test run. The
   fix is one line (`datetime.now(datetime.UTC)`), but it's not in
   this session's scope. Fold into the next CRUD-layer touch (most
   likely Session 3b when the schema gains priority / action_by).

6. **Aggrid quick-filter asymmetry.** AgGrid's column floating-filter
   searches all visible columns; `search_projects()` only searches
   name / address / job_number. Both work in the Table view (the
   floating-filter is a per-column UI nicety; the top search bar
   still routes through `search_projects()`). **Document the asymmetry
   in the Projects page user-facing help text** — not a code change,
   one paragraph.

## Risks called out by scout that DID NOT bite

- **Color palette adoption.** The scout flagged that adopting the
  prompt's palette globally meant the existing `status_badge` helper
  in `formatters.py` would shift colors for any page using it. The
  view-switcher-builder updated `formatters.py:_STATUS_COLORS["project"]`
  to match the new palette; Home.py's project chart used `pbs` for
  bar colors (not `status_badge`), so no visual regression.
- **3 expander-DOM tests in `tests/test_projects_search.py`.** All
  three were rewritten to read the new
  `st.session_state["ui:projects:_test_visible_ids"]` slot. Each
  asserts the visibility semantic ("after searching X, every visible
  job_number contains X") cleanly, without DOM inspection.
- **`st.rerun()` interaction with aggrid.** Scout #3 warned that a
  forced rerun inside the save handler eats in-flight edits.
  `handle_row_save` does NOT call `st.rerun()`; it returns the error
  string (if any) to the caller, which surfaces a toast. Aggrid
  redraws naturally on the next user action.
- **6× DB re-fetch per rerun.** Old page fetched projects once per
  status tab (6 tabs = 6 queries). New page fetches ONCE at the top of
  the render (lines 240-247 of `1_Projects.py`); all 4 views consume
  the same in-memory list. Cumulative cost is now constant in number
  of views.
- **`status_change` action type.** Scout #3 noted there's no
  `status_change` action — status changes are bundled under `updated`
  with `"status"` in details. `summarize_activity` recognizes the
  pattern explicitly and renders "Status changed to {label}".

After pulling `feature/projects-ui-uplift` and running
`py -3.14 launch_platform.py`:

1. Open `http://localhost:8502/Projects` after logging in. Confirm:
   - 5 stat cards (Total / Active / Prospects / On Hold / Completed)
     show non-zero counts against the seeded DB.
   - "Create New Project" expander collapsed by default.
   - "Search projects" form bar visible.
   - **View switcher** shows 4 radio options (Table / Kanban / Timeline / Calendar), Table selected.
   - **Status filter** segmented control to the right shows "All" + 5 status labels.

2. **Table view (Gate 3 already eyeballed by Juan)** — re-confirm:
   - AgGrid renders with columns ID hidden, Job # pinned left,
     Project / Status / Client / Address / City / County / Scope /
     Start / Target Close / Notes.
   - Status cells show colored pills (green / amber / purple / gray /
     dark-gray for active / prospect / on_hold / completed / archived).
   - Click a Project name cell → edit → click out → toast "Saved name
     for {job_number}".
   - Click a Status cell → dropdown shows the 5 status values → pick
     one → toast confirms the change.
   - Floating filter row at the top of each column responds to typed
     queries.
   - Click a row → detail panel appears below the grid with the 6
     tabs (Details / Edit / Milestones / Calculations / Documents /
     Activity).
   - Activity tab shows the just-edited row's `Status changed to {label}`
     or `Updated: {field}` lines.
   - "← Close" button returns to the grid-only view.

3. **Kanban view** — switch to Kanban:
   - 4 columns visible (active / prospect / on_hold / completed),
     archived hidden by default.
   - Each column header shows the colored pill + count in parentheses.
   - Cards show colored left border matching the column's status color,
     job number bold, name truncated to 40 chars, client + target end
     captions.
   - "Show archived" checkbox above the columns reveals the 5th
     column when toggled.
   - Per-card Status dropdown → pick a different status → toast confirms
     + card moves to the new column.
   - Per-card "View" button → detail panel below the board.

4. **Timeline view** — switch to Timeline:
   - Horizontal Gantt bars colored by status.
   - Today marker (red dashed vertical line).
   - Footer caption "N project(s) with no dates — not shown on Timeline"
     if any seeded projects have null start AND null target_end.
   - Below-chart "Open project" selectbox → choose → detail panel
     below the chart.

5. **Calendar view** — switch to Calendar:
   - `st.info("Calendar view — coming up next.")` banner.
   - Metric "Target end dates in {Month Year}" with count.
   - Confirm this is a placeholder, not a crash.

6. **Search filter regression** — back to Table view, type a job
   number (e.g. `260901`) into the search bar, press Enter:
   - Grid filters to that one project.
   - Switch to Kanban — same filter holds (only that column shows the
     card).
   - Clear search → all projects return.

7. **Status filter** — click "On Hold" in the segmented control:
   - Grid filters to only on_hold projects.
   - Switch views — filter persists.
   - Click "All" → everything returns.

8. **Persistent state** — set view = Kanban, status filter = Active,
   refresh the browser (Ctrl+R):
   - Auth gate reappears (expected — session_state resets across
     browser reload per the prompt's "DO NOT add localStorage shim"
     guardrail).
   - After re-login the page returns to defaults (Table / All). This
     is the documented behavior — localStorage persistence is a
     deferred TODO if Juan wants it.
